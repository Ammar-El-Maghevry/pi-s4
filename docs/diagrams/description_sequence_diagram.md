# وصف مخططات التسلسل (Diagrammes de Séquence)

النظام يحتوي على **10 تدفقات (Flux)** رئيسية تشرح كيفية تفاعل المستخدم مع النظام.

---

## 1. تسجيل الدخول (Authentification)

**المشاركون**: المسؤول (Admin) ← API ← CRUD ← قاعدة البيانات

**الخطوات**:
1. المسؤول يرسل بريده الإلكتروني وكلمة السر إلى `POST /api/v1/auth/login`
2. النظام يبحث عن المستخدم في قاعدة البيانات
3. إذا وُجد المستخدم، يتم التحقق من كلمة السر (مشفرة بـ bcrypt)
4. إذا كانت صحيحة → يتم إنشاء رمز JWT وإرساله إلى المسؤول
5. إذا كانت خاطئة → يتم إرجاع خطأ 401

**الهدف**: السماح للمسؤول بالدخول إلى النظام والحصول على رمز مميز (Token) لاستخدامه في الطلبات الأخرى.

---

## 2. إنشاء طالب جديد (Création d'un étudiant)

**المشاركون**: المسؤول ← API ← JWT ← CRUD ← قاعدة البيانات

**الخطوات**:
1. المسؤول يرسل بيانات الطالب (الاسم، الرقم الجامعي، ...) مع رمز JWT
2. النظام يتحقق من صحة الرمز (قد يكون منتهيًا → خطأ 401)
3. النظام يتأكد أن الرقم الجامعي غير مكرر
4. إذا كان مكررًا → خطأ 409
5. إذا كان جديدًا → يتم إدراج الطالب في قاعدة البيانات وإرجاع معلوماته

**الهدف**: إضافة طالب جديد إلى النظام.

---

## 3. تسجيل الوجه (Enrôlement facial)

**المشاركون**: المسؤول ← API ← خدمة الصور ← InsightFace ← قاعدة البيانات

**الخطوات**:
1. المسؤول يرسل صورة الطالب إلى `POST /students/{id}/photo`
2. النظام يحفظ الصورة على القرص
3. النظام يمرر الصورة إلى محرك التعرف على الوجوه InsightFace
4. **إذا لم يتم العثور على وجه** → خطأ 422
5. **إذا تم العثور على أكثر من وجه** → خطأ 422
6. **إذا تم العثور على وجه واحد بالضبط**:
   - يتم استخراج البصمة الوجهية (512 رقم)
   - يتم حفظ البصمة مع مسار الصورة في قاعدة البيانات
   - يتم إرجاع معلومات الطالب مع `has_face_embedding = true`

**الهدف**: تسجيل بصمة وجه الطالب لاستخدامها لاحقًا في التعرف التلقائي.

---

## 4. حساب الحضور (Calcul de présence)

**المشاركون**: المسؤول ← API ← خدمة الحضور ← المحرك ← قاعدة البيانات

**الخطوات**:
1. المسؤول يطلب حساب الحضور لتاريخ معين
2. لكل طالب لديه أحداث في ذلك اليوم:
   - يتم جلب أحداث الدخول/الخروج
   - يتم بناء فترات الحضور (مثلاً: 8:15 → 10:30)
   - لكل حصة دراسية:
     - يتم حساب نسبة التداخل بين فترة حضور الطالب ووقت الحصة
     - إذا النسبة ≥ 70% → حاضر (PRESENT)
     - إذا النسبة ≥ 20% → متأخر (LATE)
     - إذا أقل → غائب (ABSENT)
     - يتم حفظ النتيجة
3. يتم إرجاع تقرير بعدد الطلاب المعالجين والنتائج المكتوبة

**الهدف**: تحويل أحداث الدخول/الخروج الخام إلى نتائج حضور مفهومة (حاضر/متأخر/غائب).

---

## 5. كاميرا الهاتف + التعرف المباشر (Flux caméra téléphone)

**المشاركون**: الهاتف ← API ← WebRTC ← حلقة التعرف ← InsightFace ← قاعدة البيانات

**الخطوات**:
1. الهاتف يفتح رابط الاقتران (`/phone-camera/{token}`)
2. يعرض معلومات الكاميرا ويطلب إذن الكاميرا
3. ينشئ اتصال WebRTC عبر STUN ويبدأ البث المباشر
4. **حلقة التعرف (كل 3 ثوانٍ)**:
   - تأخذ آخر إطار فيديو من الهاتف
   - تكتشف الوجوه في الإطار عبر InsightFace
   - تقارنها بالطلاب المسجلين
   - إذا تم التعرف على طالب ولم يتم تسجيله اليوم → تسجل حدث دخول وتحسب حضوره فورًا
5. عند الضغط على "إيقاف" → يتم إغلاق الجلسة

**الهدف**: تمكين الهاتف من العمل ككاميرا مراقبة مع التعرف التلقائي على الطلاب.

---

## 6. إعداد كاميرا جديدة (Configuration d'une caméra)

**المشاركون**: المسؤول ← API ← قاعدة البيانات

**الخطوات**:
1. المسؤول يرسل بيانات الكاميرا: الاسم، النوع (IP أو هاتف)، الإعدادات
2. إذا كان النوع "هاتف" → يتم إنشاء رمز اقتران عشوائي
3. إذا كان النوع "IP" → يتم التحقق من صحة رابط المصدر
4. يتم حفظ الكاميرا في قاعدة البيانات
5. يتم إرجاع معلومات الكاميرا مع إخفاء كلمة السر في الرابط وإظهار رابط الاقتران

**الهدف**: إضافة كاميرا جديدة إلى النظام وتكوين إعداداتها.

---

## 7. اختبار اتصال الكاميرا (Test de connexion)

**المشاركون**: المسؤول ← API ← OpenCV ← قاعدة البيانات

**الخطوات**:
1. المسؤول يطلب اختبار الاتصال بكاميرا معينة
2. إذا كانت كاميرا هاتف → نجاح فوري
3. إذا كانت كاميرا IP:
   - يحاول فتح البث عبر OpenCV
   - يقرأ إطار واحد
   - إذا نجح → يعرض الأبعاد (العرض والارتفاع)
   - إذا فشل → رسالة خطأ

**الهدف**: التأكد من أن الكاميرا تعمل قبل استخدامها.

---

## 8. إرسال رابط الاقتران عبر البريد (Envoi d'email)

**المشاركون**: المسؤول ← API ← خادم SMTP

**الخطوات**:
1. المسؤول يطلب إرسال رابط الاقتران لبريد إلكتروني معين
2. إذا لم يتم تكوين البريد الإلكتروني للكاميرا → خطأ 400
3. إذا لم يتم تكوين خادم SMTP → خطأ 503
4. إذا كان كل شيء جاهزًا:
   - يتصل بخادم SMTP (STARTTLS أو TLS)
   - يرسل رسالة HTML تحتوي على رابط الاقتران
   - يعرض نتيجة النجاح أو الفشل

**الهدف**: إرسال رابط كاميرا الهاتف إلى المسؤول عبر البريد الإلكتروني.

---

## 9. إنشاء تقرير (Génération de rapport)

**المشاركون**: المسؤول ← API ← خدمة التصدير ← قاعدة البيانات

**الخطوات**:
1. المسؤول يطلب تقريرًا بمعايير: الفترة (يومي/أسبوعي/شهري)، التنسيق (CSV/Excel/PDF)
2. النظام يجمع نتائج الحضور في الفترة المحددة
3. يقوم بتجميعها حسب كل طالب (عدد مرات الحضور، التأخير، الغياب)
4. يصدرها بالتنسيق المطلوب:
   - **CSV**: نصي مفصول بفواصل
   - **Excel**: ملف إكسل بتنسيق احترافي
   - **PDF**: مستند PDF بجدول مرتب

**الهدف**: تصدير تقارير الحضور بصيغ مختلفة للطباعة أو التحليل.

---

## 10. لوحة القيادة (Tableau de bord)

**المشاركون**: المسؤول ← API ← قاعدة البيانات

**الخطوات**:
1. المسؤول يطلب ملخص لوحة القيادة
2. النظام يقوم بـ 3 استعلامات متوازية:
   - **عدد الطلاب الكلي**: COUNT(*) FROM students
   - **الحاضرون اليوم**: عدد نتائج الحضور غير "غائب" لتاريخ اليوم
   - **الأحداث الأخيرة**: آخر 10 أحداث دخول/خروج
3. يتم تجميع المعلومات وإرجاعها

**الهدف**: إعطاء المسؤول نظرة سريعة على حالة الحضور في اليوم الجاري.
